# My First ANN · הסבר תא-אחר-תא

הנוטבוק [`My_first_ANN.ipynb`](My_first_ANN.ipynb) בונה רשת נוירונים פשוטה ב-PyTorch לחיזוי ביקורות שליליות בדאטהסט [Olist](https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce). המסמך הזה מסביר כל תא — מה עושים בו, למה, וכיצד לקרוא את הפלט.

## מטרת המשימה

**חיזוי בינארי:** האם ביקורת של הזמנה תהיה שלילית (`review_score ≤ 2`) על סמך פיצ'רים ברמת ההזמנה — זמן משלוח, דמי משלוח, מחיר, מספר פריטים ותשלומים.

**המבנה הטכני:**

- שכבה נסתרת אחת עם `ReLU`.
- שכבת פלט עם **logit יחיד** (בלי `sigmoid`).
- אימון עם `BCEWithLogitsLoss` ו-`Adam`.
- השוואה מול `LogisticRegression` כ-baseline.

**דרישות מקדימות:**

```bash
pip install pandas numpy scikit-learn torch
```

הדאטהסט של Olist מורכב מ-8 קבצי CSV. הנוטבוק צורך רק ארבעה מהם: `orders`, `reviews`, `items`, `payments`.

## חלק 3 · Feature Engineering

זהו התא הכבד ביותר בנוטבוק. שלושה שלבים:

1. **סינון וחישוב תכונות זמן** — משאירים רק הזמנות ש**נמסרו** (`order_status == 'delivered'`), כי לאחרות אין תאריך משלוח בפועל. מחשבים `delivery_days` (זמן משלוח בפועל) ו-`delivery_delay_days` (השוואה לזמן המשוער — ערך שלילי = הגיע לפני הזמן).
2. **צבירה ברמת הזמנה** — `items` ו-`payments` מכילים שורה לכל פריט/תשלום, לא לכל הזמנה. משתמשים ב-`groupby('order_id').agg(...)` כדי לצבור לרמת הזמנה. מ-`reviews` צוברים ממוצע `review_score` (במקרים שיש כמה ביקורות להזמנה).
3. **חיבור, יצירת target וניקוי** — `merge` שרשרתי לטבלה אחת. יצירת עמודת `is_negative` בינארית. הסרת שורות עם `NaN` וסינון ערכי `delivery_days` בלתי סבירים (שליליים או מעל 60 יום).

**פלט מהריצה:**

```

```

95,550 הזמנות נשארו אחרי הסינון — יותר מדי בשביל להתאמן על הכל בכל epoch לאט, אבל בגודל בסיסי לרשת קטנה.

## חלק 5 · חלוקה ל-Train/Test ונרמול

שלושה עקרונות חשובים:

- **`stratify=y`** — שומר על יחס המחלקות בין ה-train ל-test. בלי זה, בגלל האי-איזון, ה-test עלול לקבל שיעור שונה של דוגמאות שליליות.
- **`StandardScaler.fit(X_train)`** — ה-scaler מותאם **רק על train**. התאמה על כל הדאטה (כולל test) יוצרת **data leakage**: הממוצע וסטיית התקן שהמודל רואה בזמן inference "פגשו" את הדוגמאות במבחן.
- **`.astype(np.float32)`** — PyTorch עובד עם `float32` כברירת מחדל. המרה מוקדמת חוסכת המרות בהמשך.

**פלט מהריצה:**

```
X_train: (76440, 6)
X_test:  (19110, 6)
negative rate train=0.126 test=0.126
```

יחס המחלקות זהה ב-train וב-test — `stratify` עשה את שלו.

- **`batch_size=128`** — פשרה סטנדרטית בין יציבות (batch גדול) למהירות עדכון (batch קטן).
- **`shuffle=True`** — קריטי לאימון תקין. בלי ערבוב, המודל היה רואה קודם את כל השורות הראשונות ואז את האחרונות — הטיה שיטתית.

**פלט מהריצה:**

```

בכל epoch המודל רואה 598 batches × 128 דוגמאות ≈ 76,440 דוגמאות — כל דאטה האימון פעם אחת.

## חלק 7 · הגדרת ה-MLP

הארכיטקטורה: `6 → 16 → 1` — כלומר 6 פיצ'רים בקלט, 16 נוירונים בשכבה הנסתרת, ו-**logit יחיד** בפלט.

**שלוש נקודות מפתח:**

- **`nn.Sequential`** — עוטף את השכבות ברצף. מקודד את "הרשת" כאובייקט אחד.
- **`nn.ReLU()`** בין השכבות — פונקציית האקטיבציה. **בלעדיה** הרשת מתקפלת למודל ליניארי אחד (מוכיחים את זה בטעות #1 בהמשך).
- **`nn.Linear(hidden, 1)`** בסוף — **אין `Sigmoid`**. הפלט הוא logit גולמי. `BCEWithLogitsLoss` יטפל בהמרה להסתברות באופן יציב נומרית.

`.squeeze(-1)` מסיר את המימד האחרון של הפלט: `(batch, 1) → (batch,)` — כדי שיתאים לצורה של ה-`target`.

**פלט מהריצה:**

```
SimpleMLP(
  (net): Sequential(
    (0): Linear(in_features=6, out_features=16, bias=True)
    (1): ReLU()
    (2): Linear(in_features=16, out_features=1, bias=True)
  )
)
trainable parameters: 129
```

**129 פרמטרים** בסך הכל: `(6×16 + 16) + (16×1 + 1) = 112 + 17 = 129`. רשת קטנה מאוד — מתאימה לגודל הבעיה.

## חלק 8 · לולאת האימון

הליבה של הנוטבוק. שני עוגנים חשובים:

**חוסר איזון:** `pos_weight` ב-`BCEWithLogitsLoss` מכפיל את המשקל של המחלקה החיובית (המחלקה הפחות נפוצה) בפונקציית ה-loss. הנוסחה `(1 - pos_rate) / pos_rate` מחזירה 6.92 — כלומר, כל טעות בזיהוי ביקורת שלילית נחשבת בערך פי 7 מטעות בזיהוי ביקורת חיובית.

**חמשת השלבים של לולאת האימון** (מתבצעים לכל `batch`):

1. `optimizer.zero_grad()` — איפוס גרדיאנטים שהצטברו מה-batch הקודם.
2. `logits = model(xb)` — **Forward pass**.
3. `loss = criterion(logits, yb)` — חישוב הטעות.
4. `loss.backward()` — **Backpropagation** — `autograd` של PyTorch מחשב את הגרדיאנטים אוטומטית.
5. `optimizer.step()` — עדכון המשקלים לפי הגרדיאנטים.

**פלט מהריצה:**

```
pos_weight = 6.92
epoch  1/15  train_loss = 1.1001
epoch  3/15  train_loss = 1.0158
epoch  6/15  train_loss = 1.0082
epoch  9/15  train_loss = 1.0050
epoch 12/15  train_loss = 1.0038
epoch 15/15  train_loss = 1.0031
```

ה-loss יורד יפה מ-1.10 ל-1.00 — התכנסות נאה. הירידה מתקשה אחרי epoch 6 (`1.008 → 1.003`) — סימן שהמודל מגיע לגבול הביצועים שלו על הפיצ'רים הקיימים.

## חלק 11 · ארבע טעויות נפוצות

הסעיף החינוכי של הנוטבוק. כל תת-סעיף מבצע **גרסה שבורה בכוונה** של המודל כדי להראות מה קורה. זה חיוני לזיהוי הבאגים כשהם קורים בעבודה אמיתית.

**פלט מהריצה:**

```

Logistic Regression:  F1=0.375  ROC-AUC=0.738

**הבדל של 0.002 ב-F1 ואפס ב-AUC** — הוכחה חיה שהרשת התקפלה למודל ליניארי. אם מישהו הראה לך רשת שמקבלת בדיוק את התוצאות של רגרסיה לוגיסטית — בדוק אם שכחת פונקציית אקטיבציה.

### טעות 2 · Learning Rate לא מכויל

סורק שלושה ערכים: `1e-1` (גבוה מדי), `1e-3` (נכון), `1e-6` (נמוך מדי).

### טעות 4 · הבאג של ה-`sigmoid` הכפול

בונים `BuggyMLP` שמוסיף `nn.Sigmoid()` בסוף השכבות, ואז משתמשים באותה `BCEWithLogitsLoss`. הבעיה: `BCEWithLogitsLoss` מכיל `sigmoid` בפנים — הפעלה כפולה = גרדיאנטים חלושים = אימון פחות יעיל. **הקוד לא זורק שגיאה. הכל "רץ".**

**פלט מהריצה:**

```

correct MLP:        F1=0.427  ROC-AUC=0.756
```

**הפער דרמטי:** F1 קרוב ל**חצי** מהמודל הנכון (0.224 vs 0.427). זו הטעות הכי חמקנית מבין הארבע — אתה רואה מספרים גרועים אבל בלי סימני שגיאה. הכלל: אם משתמשים ב-`BCEWithLogitsLoss` — אין `Sigmoid` בסוף. אם משתמשים ב-`CrossEntropyLoss` — אין `Softmax` בסוף.

# 🧪 התנסות — חלק 2

הרשת הראשונה שלנו רצה. עכשיו נבצע שלוש התנסויות שהמשימה דרשה: רגולריזציה עם Dropout ו-Weight Decay, Early Stopping, ו-Autoencoder ל-anomaly detection על Olist.

## חלק 13 · Dropout + Weight Decay

**המטרה:** להראות **בפועל** איך שני מנגנוני רגולריזציה משפיעים על הפער בין `train_loss` ל-`val_loss`. פער גדול = overfitting; מנגנון רגולריזציה טוב יצמצם את הפער בלי לפגוע ב-val_loss.

**המבנה שנבנה:**

- רשת קצת יותר עמוקה (`6 → 32 → 32 → 1`) כדי שה-overfitting יופיע מהר על דאטה קטן יחסית
- חלוקה חדשה: 10% מ-`X_train_s` נופל ל-`X_val` — לא נוגעים ב-test
- 4 גרסאות של אותו מודל:

| שם | Dropout | Weight Decay |
|---|---|---|
| baseline | 0 | 0 |
| dropout | 0.3 | 0 |
| weight_decay | 0 | 0.01 (via AdamW) |
| dropout + wd | 0.3 | 0.01 |

**מה עוקבים:** `train_loss` ו-`val_loss` בכל epoch (30 סה"כ). בסוף מייצרים טבלה + 4 גרפים המראים כמו לוחות חשמל: איפה הרגולריזציה עוזרת ואיפה לא.

**קריאה נכונה של התוצאה:**
- אם **הפער נסגר** ו-`val_loss` נשאר נמוך = הרגולריזציה עובדת ✓
- אם **הפער נסגר** אבל `val_loss` **עלה** = רגולריזציה חזקה מדי, המודל עשה underfitting ✗
- אם **הפער נשאר** למרות הרגולריזציה = צריך יותר דאטה, לא יותר רגולריזציה

## חלק 14 · Early Stopping

**המטרה:** להראות את **הפוטנציאל הכלכלי** של early stopping. בבסיס — לא צריך לרוץ 50 epochs אם המודל התכנס אחרי 15. בעולם האמת של אימונים על GPUs יקרים, זה חוסך שעות ו-$.

**המבנה:**

1. מריצים baseline רגיל ל-50 epochs וממדדים את הכי-טוב-val_loss
2. מריצים אותו baseline עם `EarlyStopping(patience=7)`
3. מדפיסים כמה epochs נחסכו והאם ה-best val_loss שנשמר תואם למה שהריצה הארוכה מצאה

**איך `EarlyStopping` עובד בפועל:**

- שומר את המשקלים הכי טובים שנראו (`best_state`)
- סופר כמה epochs עברו מאז ה-best (`counter`)
- אם `counter` הגיע ל-`patience` = עצור, החזר את המשקלים הטובים ביותר
- `min_delta=1e-4` = שיפור קטן מאוד לא נחשב "שיפור אמיתי" - מגן מ-noise

**הפתעה נפוצה:** לפעמים ה-full run ו-early stop מגיעים לאותו best val loss בדיוק — כי שניהם עוצרים במעשה באותו epoch (רק early stop לא ממשיך לרוץ חינם).

## חלק 15 · Autoencoder ל-Anomaly Detection

**המטרה:** לזהות הזמנות **חריגות** ב-Olist בלי להשתמש בתוויות בכלל. גישה: Autoencoder לומד לשחזר "הזמנה טיפוסית" — הזמנות שקשה לו לשחזר הן ה-outliers.

**המבנה של ה-Autoencoder:**

```
6 features → [encoder: 6→4→2] → 2-neuron bottleneck → [decoder: 2→4→6] → 6 features
```

הצוואר הצר של 2 נוירונים הוא הקסם: הרשת חייבת "לדחוס" כל הזמנה ל-2 מספרים ואז לשחזר אותה. הדחיסה שומרת רק את המבנה **הכי נפוץ**. חריגות = הזמנות שלא נכנסות בדפוס הזה.

**התהליך:**

1. **הכנה:** מתאימים `StandardScaler` חדש על **כל** ה-95,550 הזמנות (train + test) — אין validation set כי אין תוויות
2. **אימון:** MSE loss בין `x` ל-`reconstruction`, Adam optimizer, 30 epochs
3. **ניקוד:** לכל הזמנה מחשבים את `MSE(x, x̂)` — זו ה-`reconstruction_error`
4. **סף:** לוקחים את ה-99th percentile של השגיאות — כל הזמנה מעל הסף מסומנת כ-**anomaly**
5. **אינטרפרטציה:** מציגים את 10 החריגות הגבוהות ביותר + טבלת השוואה בין הסטטיסטיקה של "רגילות" ל-"חריגות"

**גרף הפעמון:** ה-histogram של השגיאות (עם ציר y לוגריתמי) מראה את **הצורה של הזנב**. הזנב הימני הארוך הוא בדיוק שם ה-anomalies יושבות. הקו האדום הוא סף ה-1%.

## חלק 16 · דיון — הפירוש הוא העבודה

**שאלה 1 · האם הרגולריזציה צמצמה את ה-overfitting?**

מסתכלים על עמודת `gap (val - train)` מחלק 13. אם הפער התכווץ עם Dropout + WD לעומת ה-baseline = הרגולריזציה עשתה את שלה. אבל צריך לוודא שלא הגזמנו — `best_val_loss` צריך להישאר נמוך, לא לעלות.

**דוגמה קלאסית של איך נראית תוצאה טובה:**

| variant | train | val | gap |
|---|---|---|---|
| baseline | 0.85 | 0.98 | **+0.13** ← יש overfitting |
| dropout | 0.92 | 0.95 | **+0.03** ← הרגולריזציה סגרה את הפער |
| weight_decay | 0.90 | 0.94 | **+0.04** ← גם עוזר |
| both | 0.95 | 0.96 | **+0.01** ← הכי בטוח, אבל val loss קצת עלה |

**שאלה 2 · האם ה-anomalies שה-Autoencoder סימן נראות חשודות?**

זה **הצעד הכי חשוב במחזור החיים של Data Scientist** — המודל **מציע**, האדם **מפרש**. הקוד בסוף חלק 16 עובר על 5 החריגות הגבוהות ביותר, וסורק אילו פיצ'רים חורגים מהממוצע פי 4 ומעלה. סוגי חריגות נפוצים שאמורים לצוץ:

- **`delivery_days` קרוב ל-60** — משלוחים בעייתיים במיוחד
- **`price` הרבה מעל הממוצע** — הזמנות של פריטים יקרים מאוד
- **`freight_value` גדול יחסית ל-`price`** — משלוחים לאזורים מרוחקים או פריטים כבדים
- **`payment_installments` חריג (12+)** — הזמנות עם פריסת תשלומים עמוקה במיוחד
- **שילובים** — הרבה פעמים החריגה היא לא פיצ'ר בודד אלא **צירוף מוזר** של כמה

**מה אנחנו לא יודעים מהמודל:** האם החריגות האלה הן **הונאה, שגיאת הזנה, לקוחות פרימיום, או סתם רעש**. המודל לא מבחין. **הכרעה עסקית זו העבודה שלנו**.
