מחזור החיים של פרויקט Data Science

The Data Science Lifecycle
📝 בקצרה
פרויקט Data Science מורכב משמונה שלבים עיקריים, מאורגנים במחזור איטרטיבי שלא נגמר. ההבדל בין פרויקט מצליח לפרויקט שמת בייצור הוא לרוב לא טכניקת המודל - אלא ההבנה שהתהליך הוא לולאה ולא קו ישר. החל מהגדרת הבעיה, דרך איסוף והכנת דאטה, אימון והערכת מודל, פריסה לייצור, ולבסוף ניטור וכיוונון מחדש על בסיס מציאות משתנה. עיקרון מנחה: "Business First" - מתחילים מהבעיה העסקית, לא מהמודל. כ-80% מהזמן מושקעים בעבודת דאטה (שלבים 1-3), ו-20% בלבד באימון והגשת המודל. המדריך הזה מאורגן לפי הכלים והקוד שמשרתים את השלבים האלה: Python, Pandas, SQL, ויזואליזציה, scikit-learn, PyTorch/Keras, Hugging Face, AI Agents ותקשורת תוצאות (Storytelling).
📊 המחשה ויזואלית
🔢 שמונת השלבים בפירוט
  • 1. Problem Definition - הגדרת הבעיה העסקית: מה השאלה? מי הלקוח? מה המדד להצלחה? שלב Understanding.
  • 2. Data Collection - איסוף נתונים ממקורות שונים (DBs, APIs, scraping, sensors). שלב Understanding.
  • 3. Data Preparation - ניקוי, טיפול בערכים חסרים, feature engineering, חלוקה ל-train/test. הצינור הכי יקר זמן. שלב Preparation.
  • 4. Model Development - בחירת אלגוריתם, אימון, כיוונון היפר-פרמטרים. שלב Modeling.
  • 5. Evaluation - מטריקות, cross-validation, A/B test, השוואה ל-baseline. שלב Modeling.
  • 6. Deployment · CI/CD - פריסה לסביבת production עם מנגנוני בקרת גרסה. שלב Deployment & MLOps.
  • 7. Monitoring - מעקב אחר ביצועים, drift detection, latency, עלות. שלב Deployment & MLOps.
  • 8. Re-train / Fine-tune - כאשר drift או ירידה בביצועים מתגלים, חוזרים לשלב 1 או 3 או 4. שלב Deployment & MLOps.

💡 שלושה עקרונות מנחים
  • 80% מהזמן מוקדש לעבודת דאטה (שלבים 1-3) - איסוף, ניקוי, הכנה. רק 20% להגדרת המודל. כל מי שמדלג על העבודה הזו משלם על כך פי שלוש בייצור.
  • Business First - מתחילים מהבעיה העסקית, לא מהמודל. "מה נשתפר אם המודל יעבוד מושלם? במה זה יתורגם לכסף, ללקוחות, להחלטות?" לפני שמסתכלים על דאטה - מגדירים את ה-success metric.
  • Never Ends (הלולאה) - Monitor → זיהוי Drift → Re-train/Fine-tune → חזרה למעגל. בעולם האמיתי הדאטה משתנה כל יום, ההתפלגות שעליה אימנו אינה ההתפלגות שעליה אנחנו מנבאים מחר. מודלים שלא מנוטרים מתים בשקט.
✓ למה המחזור חשוב לך
  • מאפשר להבין באיזה שלב נמצאים בכל רגע - מקשר בין עבודה יומית למפת התהליך הכוללת
  • מבהיר איזה כלי מתאים לכל שלב - SQL לאיסוף, Pandas להכנה, sklearn למודל, Plotly/Storytelling להגשה
  • נותן שפה משותפת לעבודה בצוות - PM, Data Engineer, Data Scientist ו-ML Engineer יודעים לאיזה שלב כל אחד אחראי
  • מעודד התייחסות ל-MLOps כחלק אינטגרלי, לא כאחרי-מחשבה
  • מקל על תיעוד הפרויקט - כל שלב מקבל מסמך, מטריקה ובדיקת איכות

רשימות - מבנה הנתונים הבסיסי

Lists - Indexing & Slicing
📝 בקצרה
רשימה (list) היא רצף מסודר וניתן לשינוי של אלמנטים מסוגים שונים. אינדוקס מתחיל מ-0, ופריסה (slicing) מאפשרת לקבל תת-רשימה. בסיס הכרחי לכל עיבוד נתונים לפני מעבר ל-NumPy ו-Pandas.
💻 קוד פייתון
# יצירת רשימה
x = ['a', 'b', 'c', 'd', 'e']

# בחירת איברים
x[0]      # האיבר הראשון: 'a'
x[-1]     # האיבר האחרון: 'e'
x[1:3]    # מאינדקס 1 (כולל) עד 3 (לא כולל)
x[:3]     # מההתחלה עד אינדקס 3
x[2:]     # מאינדקס 2 עד הסוף

# מתודות שימושיות
x.sort()              # מיון במקום
x.count('a')          # ספירת מופעים
len(x)                # אורך הרשימה
[1, 2] + [3, 4]       # שרשור: [1, 2, 3, 4]
💡 שימושים מהעולם האמיתי
  • אחסון רשימת שמות עמודות לבחירה דינמית ב-DataFrame
  • צבירת תוצאות בלולאת אימון מודל (loss per epoch)
  • פירוק שורת CSV לפני המרה ל-Series
  • בניית רשימת קבצים לקריאה ב-batch
  • החזקת hyperparameters לחיפוש Grid Search
⚠️ מלכודות נפוצות
  • אינדקס מתחיל מ-0 ולא מ-1, בניגוד ל-R ו-SQL - שגיאות off-by-one נפוצות
  • פריסה x[1:3] כוללת את 1 אך לא את 3 - חשוב לזכור את ה-exclusive end
  • הפעלת x.sort() משנה את הרשימה במקום ומחזירה None - אל תכתבו y = x.sort()

מילונים - מבנה key-value

Dictionaries
📝 בקצרה
מילון (dict) שומר נתונים בזוגות מפתח-ערך, כאשר המפתחות חייבים להיות ייחודיים. מבנה זה הוא הבסיס ליצירת DataFrame ולעבודה עם תגובות JSON מ-API. ניגשים לערך לפי המפתח ולא לפי מיקום.
💻 קוד פייתון
# יצירת מילון
x = {'a': 1, 'b': 2, 'c': 3}

# גישה לערכים
x['a']        # 1
x.keys()      # dict_keys(['a', 'b', 'c'])
x.values()    # dict_values([1, 2, 3])

# שימוש קלאסי - בניית DataFrame
import pandas as pd
df = pd.DataFrame({
    'a': [1, 2, 3],
    'b': [4, 5, 6],
    'c': ['x', 'x', 'y']
})
💡 שימושים מהעולם האמיתי
  • המרת תגובת JSON מ-API ל-DataFrame
  • מיפוי קטגוריות לערכים מספריים לפני אימון מודל
  • החזקת קונפיגורציה של ניסוי (learning_rate, batch_size וכו’)
  • ספירת תדירויות (counter pattern) לפני value_counts
  • החלפת ערכים ב-Series עם df[’col’].map(mapping_dict)
⚠️ מלכודות נפוצות
  • מפתחות כפולים יידרסו בשקט - הערך האחרון מנצח
  • ניסיון גישה למפתח שלא קיים זורק KeyError - השתמשו ב-dict.get(key, default)
  • ב-Python 3.7+ הסדר נשמר, אך אל תסתמכו על סדר כשמדובר בנתונים סמנטיים

NumPy arrays - חישוב וקטורי

NumPy Arrays
📝 בקצרה
NumPy מספק מערכים רב-מימדיים עם חישוב וקטורי מהיר בהרבה מלולאות פייתון. זהו המנוע מתחת ל-Pandas ול-scikit-learn. כל פעולה אריתמטית מתבצעת element-wise על כל המערך.
💻 קוד פייתון
import numpy as np

# יצירת מערך
np.array([1, 2, 3])           # מרשימה
np.arange(1, 5)               # [1, 2, 3, 4]
np.arange(1, 10, 2)           # [1, 3, 5, 7, 9] - עם צעד
np.repeat([1, 3], 3)          # [1, 1, 1, 3, 3, 3]
np.tile([1, 3], 3)            # [1, 3, 1, 3, 1, 3]

# פונקציות סטטיסטיות
np.mean(x), np.std(x), np.var(x)
np.min(x), np.max(x), np.sum(x)
np.quantile(x, 0.5)           # חציון
np.log(x), np.exp(x)          # חישובים אלמנט-בי-אלמנט
💡 שימושים מהעולם האמיתי
  • יצירת רעש (np.random) לסימולציות וצריכת מודל
  • חישוב מטריצות מאפיינים עבור מודל ML
  • חישוב מהיר של אינדיקטורים טכניים על סדרת מחירים
  • נירמול מערך תמונה לפני העברה לרשת נוירונים
  • יצירת bins לבדיקות סטטיסטיות (histograms)
⚠️ מלכודות נפוצות
  • השמה a = b לא מעתיקה - שינוי ב-a ישפיע על b. השתמשו ב-b.copy()
  • broadcasting יכול להפיק תוצאות לא צפויות כששתי צורות תואמות בטעות
  • פעולות In-place (a += 1) משנות dtype - הקפידו על dtype מראש

פעולות על מחרוזות לעיבוד טקסט

String Manipulation
📝 בקצרה
טקסט הוא חלק עצום מהנתונים בעולם האמיתי - לוגים, שמות, כתובות, תיאורי מוצר. שליטה בפעולות מחרוזת בסיסיות חוסכת זמן בניקוי נתונים לפני המעבר ל-Pandas .str accessor.
💻 קוד פייתון
s = "Jack and Jill"

# שינוי רישיות
s.upper()         # 'JACK AND JILL'
s.lower()         # 'jack and jill'
s.title()         # 'Jack And Jill'

# החלפה ופיצול
s.replace("J", "P")        # 'Pack and Pill'
"a,b,c,d".split(",")        # ['a', 'b', 'c', 'd']

# חיתוך מחרוזת
s[0:4]            # 'Jack' - בדומה לרשימה

# שרשור וכפל
"Data" + "Camp"   # 'DataCamp'
3 * "ab "         # 'ab ab ab '
💡 שימושים מהעולם האמיתי
  • סטנדרטיזציה של שמות לקוחות לפני join (lower + strip)
  • ניקוי כתובות אימייל לאימות תקינות
  • פיצול שדה שם מלא לשם פרטי ומשפחה
  • חילוץ קוד מדינה מקידומת טלפון
  • ניקוי תיאורי מוצר מ-HTML ורווחים מיותרים
⚠️ מלכודות נפוצות
  • מחרוזות הן immutable - s.replace() מחזיר חדשה ולא משנה את המקור
  • split ללא ארגומנט מפצל לפי whitespace, עם "" זורק שגיאה
  • שכחת לטפל ב-case sensitivity לפני השוואה (Use vs use)

pd.merge(adf, bdf, how='right', on='x1')   # שומר את כל bdf
pd.merge(adf, bdf, how='inner', on='x1')   # רק התאמות
pd.merge(adf, bdf, how='outer', on='x1')   # איחוד מלא

# מיזוג עם indicator לבדיקה
pd.merge(adf, bdf, how='outer', indicator=True)

pd.concat([df1, df2], axis='columns')      # אופקי

סוגי JOIN בסיסיים

INNER, LEFT, RIGHT, FULL JOIN
📝 בקצרה
JOIN משלב שורות משתי טבלאות לפי שדה מקשר משותף. INNER מחזיר רק שורות עם התאמה בשתי הטבלאות, LEFT שומר את כל השורות מהטבלה השמאלית, RIGHT שומר את כל השורות מהטבלה הימנית ו-FULL OUTER שומר את כולן. בחירת סוג ה-JOIN משפיעה ישירות על כמות הנתונים שתקבלו ועל הטיפול ב-NULL.
💻 קוד פייתון
-- INNER JOIN: רק התאמות בשתי הטבלאות
SELECT *
FROM artist AS art
INNER JOIN album AS alb
  ON art.artist_id = alb.artist_id;

-- LEFT JOIN: כל האמנים, גם בלי אלבום
SELECT *
FROM artist AS art
LEFT JOIN album AS alb
  ON art.artist_id = alb.artist_id;

-- FULL OUTER JOIN: כל השורות משתי הטבלאות
SELECT *
FROM artist AS art
FULL OUTER JOIN album AS alb
  ON art.artist_id = alb.artist_id;

📝 בקצרה

Semi Join ו-Anti Join עם תתי-שאילתות

Semi & Anti joins via subqueries
📝 בקצרה
Semi Join בוחר שורות מהטבלה הראשונה רק כאשר תנאי מתקיים בטבלה השנייה, באמצעות WHERE … IN (subquery). Anti Join עושה את ההפך עם NOT IN — מחזיר רק שורות ללא התאמה. שתי הטכניקות יעילות לסינון על בסיס קיום בטבלה אחרת בלי לגרום לכפילויות כמו ב-JOIN רגיל.
💻 קוד פייתון
-- Semi Join: אלבומים של אמנים שקיימים בטבלת artist
SELECT *
FROM album
WHERE artist_id IN (
  SELECT artist_id FROM artist
);

-- Anti Join: אלבומים שאין להם אמן בטבלת artist
SELECT *
FROM album
WHERE artist_id NOT IN (
  SELECT artist_id
  FROM artist
  WHERE artist_id IS NOT NULL
);

-- חלופה מומלצת ב-Anti Join: NOT EXISTS
SELECT *
FROM album a
WHERE NOT EXISTS (
  SELECT 1 FROM artist ar
  WHERE ar.artist_id = a.artist_id
);
💡 שימושים מהעולם האמיתי
  • מציאת מוצרים שנמכרו לפחות פעם אחת (Semi Join)
  • איתור משתמשים שמעולם לא הזמינו (Anti Join)
  • סינון לידים שכבר הומרו ללקוחות לפני קמפיין שיווקי
  • זיהוי שורות יתומות לפני ניקוי dataset לאימון מודל
  • בידוד פיצ’רים שלא נכללים ברשימת black-list
⚠️ מלכודות נפוצות
  • NOT IN עם NULL בתת-שאילתה מחזיר תמיד 0 שורות — תמיד הוסיפו IS NOT NULL
  • Semi/Anti Join דרך IN לא מכפיל שורות, בניגוד ל-INNER JOIN על מפתח לא ייחודי
  • NOT EXISTS לרוב מהיר יותר מ-NOT IN ובטוח יותר מול NULL

Pipeline ו-ColumnTransformer

Pipeline + ColumnTransformer
📝 בקצרה
Pipeline משרשרת שלבי עיבוד מקדים ומודל לאובייקט אחד עם fit/predict, ומונעת data leakage ב-cross-validation. ColumnTransformer מאפשר להחיל טרנספורמציות שונות על עמודות שונות — סקיילינג למספריים ו-OneHot לקטגוריאליים. זהו הסטנדרט בעבודה מקצועית.
💻 קוד פייתון
from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression

numeric_cols = ['age', 'income']
cat_cols = ['country', 'gender']

numeric_pipe = Pipeline([
    ('imputer', SimpleImputer(strategy='median')),
    ('scaler',  StandardScaler()),
])

cat_pipe = Pipeline([
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('onehot',  OneHotEncoder(handle_unknown='ignore')),
])

preprocess = ColumnTransformer([
    ('num', numeric_pipe, numeric_cols),
    ('cat', cat_pipe,     cat_cols),
])

model = Pipeline([
    ('prep', preprocess),
    ('clf',  LogisticRegression(max_iter=1000)),
])

model.fit(X_train, y_train)
💡 שימושים מהעולם האמיתי
  • מודלים על נתונים טבלאיים מעורבים (מספריים + קטגוריאליים)
  • מניעת data leakage בעת שימוש ב-cross_val_score ו-GridSearchCV
  • הפצת מודל אחד מסודר ל-production בלי לשכוח שלבים
  • השוואת מודלים שונים עם אותו preprocessing זהה
  • טיפול עקבי בערכים חסרים במהלך אימון וב-inference
⚠️ מלכודות נפוצות
  • fit_transform על test set שובר את הלוגיקה — תמיד fit על train בלבד
  • OneHotEncoder ללא handle_unknown='ignore' יקרוס על קטגוריות חדשות ב-production
  • ColumnTransformer עם remainder='drop' (ברירת המחדל) משמיט עמודות שלא ציינתם

GridSearchCV ו-RandomizedSearchCV

Hyperparameter tuning
📝 בקצרה
GridSearchCV סורק את כל הצירופים בגריד פרמטרים עם cross-validation ובוחר את הטוב ביותר. RandomizedSearchCV מדגם תת-קבוצה אקראית — מהיר משמעותית כשגריד גדול. שני הכלים עוזרים לבחור היפרפרמטרים אופטימליים בלי לדגום ידנית ומדווחים best_score_ ו-best_estimator_.
💻 קוד פייתון
import numpy as np
from sklearn.model_selection import GridSearchCV, RandomizedSearchCV
from sklearn.neighbors import KNeighborsClassifier

knn = KNeighborsClassifier()

# חיפוש ממצה
params = {
    'n_neighbors': np.arange(1, 10),
    'metric':      ['euclidean', 'cityblock'],
}
grid = GridSearchCV(estimator=knn, param_grid=params, cv=5)
grid.fit(X_train, y_train)
print(grid.best_score_)
print(grid.best_estimator_.n_neighbors)

# חיפוש אקראי — מהיר יותר
rand_params = {
    'n_neighbors': range(1, 50),
    'weights':     ['uniform', 'distance'],
}
rsearch = RandomizedSearchCV(
    estimator=knn,
    param_distributions=rand_params,
    cv=4, n_iter=8, random_state=5,
)
rsearch.fit(X_train, y_train)
print(rsearch.best_score_)
💡 שימושים מהעולם האמיתי
  • כיוון max_depth ו-n_estimators ב-RandomForest/XGBoost
  • מציאת C ו-gamma אופטימליים ל-SVM
  • בחירת alpha ב-Ridge/Lasso
  • אופטימיזציה של k ב-KNN למערכות המלצה
  • סריקת היפרפרמטרים של Pipeline שלם כולל preprocessing
⚠️ מלכודות נפוצות
  • Grid גדול מדי = זמן ריצה אקספוננציאלי — התחילו עם RandomizedSearch
  • scoring ברירת המחדל הוא accuracy — על dataset לא מאוזן הגדירו f1/roc_auc במפורש
  • ללא Pipeline, סקיילינג מחוץ ל-CV יוצר דליפת מידע בכל קיפול

הערכת מודלים ו-Cross-Validation

Model evaluation metrics
📝 בקצרה
הערכת מודל לעולם לא מסתמכת על מספר יחיד. לסיווג: accuracy, precision/recall/f1, confusion matrix ו-classification_report. לרגרסיה: MAE, MSE/RMSE ו-R². ל-clustering: ARI, homogeneity, V-measure. cross_val_score נותן הערכה יציבה יותר מ-split יחיד על ידי ממוצע על כמה קיפולים.

from sklearn.metrics import (
    accuracy_score, classification_report, confusion_matrix,
    mean_absolute_error, mean_squared_error, r2_score,
)
from sklearn.model_selection import cross_val_score

# סיווג
print(accuracy_score(y_test, y_pred))
print(confusion_matrix(y_test, y_pred))
print(classification_report(y_test, y_pred))

# רגרסיה
print(mean_absolute_error(y_true, y_pred))
print(mean_squared_error(y_true, y_pred))
print(r2_score(y_true, y_pred))

# Cross-validation: ממוצע יציב
scores = cross_val_score(knn, X_train, y_train, cv=4)
print(scores.mean(), scores.std())
💡 שימושים מהעולם האמיתי

  • ניטור drift של מודל בייצור מול baseline היסטורי
  • אופטימיזציה של threshold סיווג על בסיס precision-recall curve
⚠️ מלכודות נפוצות
  • accuracy על dataset לא מאוזן (95% מחלקה אחת) חסר משמעות — השתמשו ב-f1 או ROC-AUC
  • MSE רגיש מאוד לערכי קצה — שקלו MAE כשיש outliers
  • cross_val_score על נתוני סדרת זמן ב-KFold שובר את סדר הזמן — השתמשו ב-TimeSeriesSplit

התפלגויות

Distribution Charts
📝 בקצרה
להבנת איך משתנה מספרי מתפזר משתמשים ב-histogram, box plot, violin plot ו-density plot. Histogram מציג תדירות לפי binים; Box plot מציג חמש סטטיסטיקות סיכום וחריגים; Violin משלב את שניהם; Density יוצר עקומה חלקה לזיהוי שיאים מרובים (multimodal).
💻 קוד פייתון
מתי להשתמש בכל אחד?
• Histogram    → סקירה ראשונית של התפלגות שכר/גיל (משתנה אחד)
• Box plot     → השוואת התפלגויות בין 2-10 קבוצות (יעילות דלק לפי יצרן)
• Violin plot  → כאשר חשוב לראות את צורת ההתפלגות (bimodality)
• Density (KDE) → השוואה חלקה של מספר התפלגויות חופפות
• Strip / Swarm → דאטה קטן (<200 נקודות) - הצגת כל נקודה

כלל: התחילו ב-histogram להבנה, ועברו ל-box/violin להשוואה.
💡 שימושים מהעולם האמיתי
  • Histogram: התפלגות שכר בארגון לאיתור פערים
  • Box plot: השוואת זמני קריאה לפי קבוצות גיל
  • Violin: בדיקת השפעת מינון תרופה (לחיפוש bimodality)
  • Density: השוואת מחירי דירות בכמה ערים על אותו ציר
  • Box plot ב-EDA: זיהוי מהיר של חריגים לפני בניית מודל
⚠️ מלכודות נפוצות
  • Histogram רגיש מאוד למספר ה-bins - מעט מדי מסתיר את הצורה, יותר מדי מרעיש
  • Box plot מסתיר התפלגות bimodal - נראית סימטרית גם כשיש שתי פסגות
  • Violin שטוח עלול להטעות עבור מדגמים קטנים מאוד (פחות מ-30 תצפיות)

שלושת מרכיבי ה-Data Storytelling

Data, Visuals & Narrative
📝 בקצרה
Data Storytelling מוגדר כשילוב של שלושה רכיבים: נתונים (Data), ויזואליזציות (Visuals) ונרטיב (Narrative). שילוב נכון של השלושה הוא מה שמייצר change - הסקת מסקנה אינה מספיקה, צריך גם להעביר אותה כך שתניע פעולה. זה הקילומטר האחרון של אנליטיקה.
💻 קוד פייתון
מסגרת לבדיקה עצמית לפני כל מצגת:

1. Data       - האם המספרים נכונים, מייצגים, ומעודכנים?
              - האם בחרתי מדדים שמשרתים את הנרטיב?

2. Visuals    - האם בחרתי את סוג הגרף המתאים לסיפור?
              - האם הסרתי clutter? (קווי רשת, מסגרות, legend מיותר)
              - האם הצבעים מדגישים את הנקודה החשובה?

3. Narrative  - מהי השאלה שאני עונה עליה?
              - מהי הפעולה שאני מבקש מהקהל לבצע?
              - האם יש arc: מצב התחלתי → קונפליקט → פתרון/CTA?
💡 שימושים מהעולם האמיתי
  • הצגת בקשת תקציב להנהלה - נתונים + גרף + סיפור הצורך = החלטה
  • דוח עוקבים (Quarterly review) - הצגת ביצועים עם נרטיב מה השתנה והבא
  • מצגת ל-stakeholders על מודל ML - הסבר השפעה עסקית עם נתונים
  • Post-mortem על ירידת KPI - story-driven עם DATA+CTA
  • פיץ’ למשקיעים - נתוני שוק + ויזואליות + סיפור החזון
⚠️ מלכודות נפוצות
  • התמקדות רק ב-Data מבלי נרטיב - הקהל יוצא ללא מסקנה ברורה
  • ויזואליות מרשימה ללא נתונים אמינים = ויברס פלוס מספרים - אובדן אמון
  • נרטיב חזק שאינו נתמך על ידי הנתונים = manipulation, פוגע באמינות לטווח הארוך

בחירת הגרף הנכון לסיפור

Choosing the Right Visualization
📝 בקצרה
כל סוג גרף מותאם לסיפור אחר: line למגמה, bar להשוואה, scatter לקשר, histogram להתפלגות. בחירה לא נכונה מקשה על הקהל להבין את הנקודה ומחלישה את המסר. תמיד התחילו מהשאלה הניהולית: מה אני רוצה שיראו? - הסוג צריך לשרת את התשובה.
💻 קוד פייתון
מטריצת החלטה מהירה:

שאלה שאני שואל              →  סוג הגרף המומלץ
----------------------------  -----------------
'איך זה השתנה לאורך זמן?'    →  Line / Area
'מי הכי גדול?'                →  Bar (ממוין!)
'האם יש קשר בין X ל-Y?'      →  Scatter (+trendline)
'איך זה מתפזר?'               →  Histogram / Box plot
'מתוך מה זה מורכב?'           →  Treemap / Stacked bar
'איפה במפה?'                   →  Choropleth / Bubble map
'איך זורם הערך בין שלבים?'    →  Sankey diagram
'מהו ה-KPI הנוכחי?'           →  Big Number / Card

כלל זהב: אם צריך להסביר איך לקרוא את הגרף - זה הגרף הלא נכון.
💡 שימושים מהעולם האמיתי
  • מנהל מבקש לדעת מי המוצר הטוב ביותר השנה → bar chart ממוין יורד
  • תיאור מגמת רישום משתמשים → line chart פשוט עם annotation לאירוע
  • הצגת קשר השכלה-שכר → scatter עם trendline ו-R²
  • Status report של אי-שוויון תשלום → box plot לפי מחלקה
  • מצגת על Funnel המרה → Sankey diagram שמראה איפה נושרים
⚠️ מלכודות נפוצות
  • שימוש ב-pie chart לרבים מ-5 קטגוריות - הופך לבלתי-קריא
  • Dual-axis chart כשהיחידות שונות - הקהל נוטה להניח קורלציה שלא קיימת
  • 3D charts מרשימים אסטטית אך מעוותים פרופורציות - הימנעו

פישוט והפחתת רעש

Decluttering & Simplifying
📝 בקצרה
עורכים אכזרית - זה הציטוט המרכזי של Cole Nussbaumer Knaflic. הסרת אלמנטים שאינם מייצגים נתונים (gridlines, borders, legend מיותר) מאפשרת לקהל להתמקד במסר. השתמשו ב-text וב-color בכוונה כדי להדגיש את הנקודה החשובה, לא כקישוט.
💻 קוד פייתון
Checklist לפישוט גרף:

☐ הסירו מסגרת הגרף
☐ הסירו/הקלישו קווי רשת
☐ נקו תוויות צירים (אל תכפילו יחידות)
☐ סימנו נתונים ישירות במקום legend נפרד
☐ הסירו markers מיותרים בקווים
☐ השתמשו ב-bold/underline בחיסכון
☐ צבע אפור לכל הנתונים, צבע בולט לעיקר
☐ כותרת = המסקנה, לא תיאור הגרף
     ('המכירות עלו 23%' - לא 'מכירות לפי חודש')

התאמת תוכן לקהל:
• Executive   → outcome & decision (KPI אחד גדול)
• Data leader → rigor & methodology
• Business    → impact על workflow היומי
💡 שימושים מהעולם האמיתי
  • ניקוי דשבורד מנהלים - הסרת gridlines וצמצום ל-3 KPI עיקריים

nn.Module, Linear Layers and Sequential
📝 בקצרה
מודולי nn מהווים את אבני הבניין של רשתות נוירונים ב-PyTorch - שכבות Linear, Dropout ופונקציות אקטיבציה. nn.Sequential מאפשר לבנות במהירות מודל פיד-פורוורד על ידי שרשור שכבות. משתמשים בו כשהארכיטקטורה ליניארית וברורה ואין צורך בלוגיקה מותאמת אישית.

from torch import nn

# שכבה ליניארית: m קלטים, n פלטים
lnr = nn.Linear(m, n)
lnr.weight   # מטריצת משקלים
lnr.bias     # וקטור הטיה

model = nn.Sequential(

    nn.ReLU(),
    nn.Linear(32, n_classes),
    nn.Softmax(dim=-1)   # שכבת אקטיבציה אחרונה

פונקציות אקטיבציה

Activation Functions
📝 בקצרה
פונקציות אקטיבציה מכניסות אי-ליניאריות לרשת ומאפשרות לה ללמוד יחסים מורכבים. בחירת האקטיבציה משפיעה על מהירות האימון, על בעיית הסיווה (saturation) ועל יציבות הגרדיאנטים. ReLU היא ברירת המחדל לשכבות נסתרות, Sigmoid לסיווג בינארי ו-Softmax לסיווג רב-מחלקתי.
💻 קוד פייתון
from torch import nn

# סיווג בינארי - פלט בין 0 ל-1
nn.Sigmoid()

# סיווג רב-מחלקתי - התפלגות הסתברויות
nn.Softmax(dim=-1)

# ברירת מחדל לשכבות נסתרות, מונעת saturation
nn.ReLU()

# גרסה שמטפטפת גם לערכים שליליים
nn.LeakyReLU(negative_slope=0.05)

# רגולריזציה: כיבוי אקראי של נוירונים
nn.Dropout(p=0.5)
💡 שימושים מהעולם האמיתי
  • ReLU בכל שכבה נסתרת של MLP או CNN סטנדרטיים
  • Sigmoid בשכבת הפלט של מסווג ספאם
  • Softmax בשכבת הפלט של מסווג ספרות MNIST
  • LeakyReLU במודלים גנרטיביים (GAN) למניעת dying ReLU
  • Dropout בין שכבות צפופות ברשת עם נטייה לאוברפיטינג
⚠️ מלכודות נפוצות
  • ReLU עלולה לגרום ל-dead neurons (פלט קבוע 0) אם learning rate גבוה מדי - LeakyReLU מקלה
  • Softmax מצריך לציין dim נכון; ב-dim שגוי הפלט לא יהיה התפלגות חוקית
  • שילוב Sigmoid + BCELoss רגיש לאי-יציבות נומרית; BCEWithLogitsLoss בטוח יותר

לולאת אימון, loss ואופטימייזרים

Training Loop, Loss and Optimizers
📝 בקצרה
לולאת האימון ב-PyTorch כתובה ידנית: forward pass, חישוב loss, backward pass ועדכון משקלים. בחירת פונקציית ה-loss תלויה במשימה (MSE לרגרסיה, CrossEntropy לסיווג), והאופטימייזר (SGD, Adam) שולט באופן עדכון הפרמטרים. שליטה בלולאה היא יסוד מרכזי שמבדיל את PyTorch מ-Keras.
💻 קוד פייתון
import torch.optim as optim
from torch import nn

loss_criterion = nn.MSELoss()
optimizer = optim.SGD(model.parameters(), lr=0.01, momentum=0.95)

model.train()                       # מצב אימון
for features, targets in dataloader:
    optimizer.zero_grad()           # איפוס גרדיאנטים
    predictions = model(features)   # forward
    loss = loss_criterion(predictions, targets)
    loss.backward()                 # backpropagation
    optimizer.step()                # עדכון משקלים
💡 שימושים מהעולם האמיתי
  • אימון מודל רגרסיה עם MSELoss על מחירי נדל"ן
  • אימון מסווג רב-מחלקתי עם CrossEntropyLoss על תמונות
  • סיווג בינארי של תמלילי סנטימנט עם BCELoss
  • כיוונון עדין של מודל מאומן מראש עם learning rate נמוך
  • רגרסיה עמידה לחריגים עם SmoothL1Loss
⚠️ מלכודות נפוצות
  • שכחת optimizer.zero_grad() גורמת לצבירת גרדיאנטים בין batches ולאימון שגוי
  • lr גבוה מדי גורם להתבדרות, נמוך מדי מאריך את האימון או נתקע במינימום מקומי
  • CrossEntropyLoss מצפה ל-logits גולמיים ולתוויות שלמות (int), לא ל-one-hot ולא לפלט Softmax

📝 בקצרה
TensorDataset ו-DataLoader אחראים על אספקת הנתונים למודל ב-batches, עם ערבוב (shuffle) ועיבוד מקבילי. הם מפרידים בין לוגיקת הנתונים ללוגיקת האימון ומאפשרים לטפל ביעילות בדאטה גדול שלא נכנס לזיכרון. זהו רכיב סטנדרטי כמעט בכל פרויקט PyTorch.

import torch

מצב הערכה ומדדים

Evaluation Mode and Metrics
📝 בקצרה
במצב הערכה מבטלים Dropout ומקפיאים סטטיסטיקות של BatchNorm כדי לקבל תוצאה דטרמיניסטית. torchmetrics מספקת מימושים מהירים של מדדים נפוצים כמו Accuracy, Precision ו-Recall שמצטברים על פני batches. תמיד צריך להעריך תחת torch.no_grad() כדי לחסוך זיכרון.
💻 קוד פייתון
import torchmetrics

model.eval()   # מצב הערכה - מבטל Dropout
metric = torchmetrics.Accuracy(
    task="multiclass", num_classes=3
)

with torch.no_grad():
    for features, targets in val_loader:
        output = model(features)
        # עדכון המדד לפי batch
        metric(output.argmax(dim=-1), targets)

accuracy = metric.compute()
print(f"Accuracy: {accuracy}")
metric.reset()   # איפוס לאפוק הבא
💡 שימושים מהעולם האמיתי
  • הערכת מסווג תמונות על סט ולידציה בסוף כל epoch
  • מעקב אחר Precision/Recall במסווג רפואי

  • השוואת שתי גרסאות מודל באותו מדד אחיד
  • ניטור מדדים תוך כדי אימון לזיהוי overfitting
⚠️ מלכודות נפוצות
  • שכחה לקרוא ל-model.eval() - Dropout ימשיך לפעול וייצור תוצאות לא יציבות

  • אי-איפוס metric בין epochs - הצבירה תכלול ערכים ישנים ותעוות את התוצאה

Sequential API של Keras

Keras Sequential Model
📝 בקצרה
Sequential הוא ה-API הפשוט ביותר ב-Keras לבניית רשת שבה כל שכבה מקבלת קלט יחיד ומחזירה פלט יחיד. מוסיפים שכבות בזו אחר זו עם .add() עד שהמודל מוכן ל-compile. מתאים לרוב מודלי MLP, CNN ו-RNN הסטנדרטיים.

import numpy as np
from tensorflow.keras.models import Sequential

data = np.random.random((1000, 100))
labels = np.random.randint(2, size=(1000, 1))

model = Sequential()
model.add(Dense(32, activation='relu', input_dim=100))
model.add(Dense(1, activation='sigmoid'))

model.compile(optimizer='rmsprop',
              loss='binary_crossentropy',
              metrics=['accuracy'])
model.fit(data, labels, epochs=10, batch_size=32)
💡 שימושים מהעולם האמיתי
  • אבטיפוס מהיר של MLP לסיווג בינארי על דאטה טבלאי

  • מודל בסיס (baseline) להשוואה מול ארכיטקטורות מורכבות יותר
⚠️ מלכודות נפוצות
  • Sequential לא תומך במספר קלטים/פלטים או בקשרי skip - צריך לעבור ל-Functional API

שכבות, אקטיבציות ורגולריזציה

Layers, Activations and Regularization
📝 בקצרה
Keras מספקת ספרייה עשירה של שכבות מוכנות: Dense לחיבור מלא, Conv2D ו-MaxPooling2D ל-CNN, Embedding ו-LSTM ל-RNN, ו-Dropout לרגולריזציה. בחירת השכבות והאקטיבציות הנכונה משקפת את סוג הנתונים: תמונות, רצפים או טבלאות.
💻 קוד פייתון
from tensorflow.keras.models import Sequential
from tensorflow.keras.layers import (
    Dense, Dropout, Conv2D,
    MaxPooling2D, Flatten, Embedding, LSTM
)

# CNN לסיווג תמונות
cnn = Sequential()
cnn.add(Conv2D(32, (3, 3), padding='same',
               activation='relu',
               input_shape=(32, 32, 3)))
cnn.add(MaxPooling2D(pool_size=(2, 2)))
cnn.add(Dropout(0.25))
cnn.add(Flatten())
cnn.add(Dense(10, activation='softmax'))

# RNN לסיווג טקסט
rnn = Sequential()
rnn.add(Embedding(20000, 128))
rnn.add(LSTM(128, dropout=0.2, recurrent_dropout=0.2))
rnn.add(Dense(1, activation='sigmoid'))
💡 שימושים מהעולם האמיתי
  • סיווג CIFAR-10 עם רשת Conv2D + MaxPooling + Dropout

מהו סוכן AI

What Is an AI Agent
📝 בקצרה
סוכן AI הוא מערכת שמשתמשת במודל שפה כדי להשיג מטרה מוגדרת על ידי המשתמש, באמצעות חשיבה, תכנון וביצוע פעולות באמצעות כלים חיצוניים. סוכן מורכב משלושה רכיבים: מודל שפה כמנוע החלטה, כלים לאינטראקציה עם העולם, ושכבת תזמור (orchestration) שמנהלת את התהליך. משתמשים בסוכנים כשהמשימה דורשת מספר צעדים, גישה למידע חיצוני או קבלת החלטות דינמית.
💻 קוד פייתון
# פסאודו-קוד של מבנה סוכן בסיסי
class AIAgent:
    def __init__(self, llm, tools, orchestrator):
        self.llm = llm                    # מנוע הנמקה
        self.tools = tools                # רשימת כלים זמינים
        self.orchestrator = orchestrator  # שכבת תזמור

    def run(self, user_goal):
        state = {"goal": user_goal, "history": []}
        while not self.orchestrator.is_done(state):
            thought = self.llm.reason(state)
            action = self.orchestrator.plan(thought, self.tools)
            result = action.execute()
            state["history"].append((thought, action, result))
        return state["history"][-1][2]
💡 שימושים מהעולם האמיתי

שכבת תזמור עם CoT ו-ReAct

Orchestration with CoT and ReAct
📝 בקצרה
שכבת התזמור מנהלת את מחזור החיים של הסוכן: זיכרון, מצב, הנמקה ותכנון. Chain-of-Thought מפרק בעיות מורכבות לצעדי חשיבה רציפים, Tree-of-Thoughts מאפשר לחקור מספר שרשרות הנמקה במקביל, ו-ReAct משלב בין Reasoning ל-Action בלולאה. בוחרים את הדפוס לפי מורכבות המשימה ולפי כמות הכלים שהסוכן צריך להפעיל.
💻 קוד פייתון
# דפוס ReAct פשוט: Thought -> Action -> Observation
prompt_template = """
Goal: {goal}

Thought: I need to figure out what to do next.
Action: search_web(query="...")
Observation: {tool_result}

Thought: Based on the result, I should...
Action: {next_action}
"""

def react_loop(goal, max_steps=5):
    history = []
    for step in range(max_steps):
        thought = llm.generate(f"Goal: {goal}\nHistory: {history}")
        action = parse_action(thought)
        if action.name == "finish":
            return action.answer
        observation = action.execute()
        history.append((thought, action, observation))

  • סוכן הוכחה מתמטית שבוחן מספר נתיבים עם ToT
⚠️ מלכודות נפוצות
  • CoT מאריך תשובות ומגדיל עלות tokens, לא תמיד שווה למשימות פשוטות

# פסאודו-קוד של דפוס Manager עם סוכני משנה
class ManagerAgent:
    def __init__(self, specialists):
        self.specialists = specialists  # dict של שם: סוכן

    def handle(self, task):
        # הסוכן המנהל בוחר מומחה דרך LLM
        decision = llm.route(task, list(self.specialists))
        specialist = self.specialists[decision.agent]
        return specialist.run(decision.subtask)

# שימוש: סוכן מחקר + סוכן דוחות
manager = ManagerAgent({
    "research": ResearchAgent(),
    "report":   ReportAgent()
})
manager.handle("Compile a Q3 market report")

RAG מול סוכני AI

RAG vs AI Agents
📝 בקצרה
RAG (Retrieval-Augmented Generation) שולף מידע מ-data store ומזין אותו ל-LLM בקריאה בודדת, וזה מתאים למענה על שאלות מתוך קורפוס ידוע. סוכן לעומת זאת מבצע מספר צעדים, מחליט דינמית באילו כלים להשתמש, ויכול לכלול בתוכו גם RAG ככלי. בוחרים ב-RAG כשהתשובה זמינה במסמכים, ובסוכן כשצריך לתכנן, להחליט ולפעול במספר שלבים.
💻 קוד פייתון
# RAG פשוט: retrieve -> generate בקריאה אחת
def rag_answer(question, vector_store, llm):
    docs = vector_store.similarity_search(question, k=4)
    context = "\n".join(d.content for d in docs)
    prompt = f"Context:\n{context}\n\nQuestion: {question}"
    return llm.generate(prompt)

# סוכן יכול להחזיק RAG ככלי בין שאר הכלים
tools = [
    {"name": "search_docs",   "fn": rag_answer},
    {"name": "send_email",    "fn": send_email},
    {"name": "create_ticket", "fn": create_ticket}
]
agent.run("Find policy on refunds and email the customer", tools)

Agent Building Frameworks
📝 בקצרה
ה-frameworks הנפוצים מספקים מבנים מוכנים לבניית סוכנים: LangChain ל-pipelines פשוטים, LangGraph לסוכנים stateful עם גרפים מורכבים, CrewAI לתיאום בין סוכנים, ו-Smolagents/LlamaIndex להתמחויות. בנוסף ישנם one-prompt agents, coding agents (Codex, Devin), ו-workflow agents ויזואליים (n8n, Make, Dify). הבחירה תלויה במורכבות, בכמות הקסטומיזציה הנדרשת וברמת הקוד שאתם מוכנים לכתוב.

# דוגמת LangGraph: סוכן עם גרף מצבים
from langgraph.graph import StateGraph, END

def call_model(state):
    response = llm.invoke(state["messages"])
    return {"messages": state["messages"] + [response]}

def should_continue(state):
    last = state["messages"][-1]
    return "tools" if last.tool_calls else END

graph = StateGraph(dict)
graph.add_node("agent", call_model)
graph.add_node("tools", tool_executor)
graph.add_conditional_edges("agent", should_continue)
graph.add_edge("tools", "agent")
graph.set_entry_point("agent")
agent = graph.compile()
